## Key Design Principles

### Vocabularies are for users, not for ontologists
Our controlled vocabularies are interfaces for end users â€” people
browsing for exercises, filtering by muscle group, building programs.
Anatomical or ontological purity is secondary to usability and
discoverability. Prefer colloquial terms over formal anatomical names
where both are accurate. When in doubt: would a gym-goer understand
this?

### Every design decision gets an ADR
`DECISIONS.md` is the institutional memory of this project. Every
non-trivial decision â€” vocabulary changes, pipeline behavior, prompt
rules â€” gets documented with rationale. ADRs are numbered sequentially.
Always write the ADR before or immediately after making a change.

### Vocabulary changes require version bumps
All ontology files carry `owl:versionInfo`. Bump independently:
- MAJOR: breaking changes (removing concepts, renaming URIs)
- MINOR: additive changes (new concepts, new properties)
- PATCH: non-breaking corrections (comments, labels)

## Hard Rules

- **Never add or remove vocabulary concepts without an ADR.** This includes muscles, movement patterns, training modalities, and involvement degrees.
- **Never modify `sources/free-exercise-db/raw/exercises.json`.** It is the upstream source dataset. Use `fetch.py` to refresh it.
- **Never bump a vocabulary version without documenting the change** in DECISIONS.md.
- **Discuss before building.** For any ontology or pipeline decision that isn't obviously mechanical, raise the trade-off first.

## Ontology Overview

**Core classes:** `feg:Exercise`, `feg:MuscleInvolvement`,
`feg:Muscle` (with subclasses `MuscleRegion`, `MuscleGroup`,
`MuscleHead`), `feg:MovementPattern`, `feg:TrainingModality`,
`feg:InvolvementDegree`, `feg:Equipment`

**Muscle hierarchy:** 3-level SKOS hierarchy (region â†’ group â†’ head).
Colloquial names preferred (e.g. "Calves" as a subregion under
"LowerLeg"). Full concept list in `muscles.ttl`.

**Movement patterns:** Flat SKOS vocabulary with some narrower
concepts. Full list in `movement_patterns.ttl`.

**Training modalities:** Assigned only when modality is a defining
characteristic. Most resistance exercises get none (ADR-021).

**Namespace:** `https://placeholder.url#` (feg:) â€” placeholder,
to be replaced before any public release.

## Pipeline Overview

### fetch.py â€” download upstream source (run before map.py on a fresh clone)
```bash
python3 sources/free-exercise-db/fetch.py
```

### preprocess.py â€” normalise source data
Reads crosswalk CSVs and replaces muscle source strings with `feg_local_name`
equivalents. Writes `exercises_normalized.json` (gitignored).

```bash
python3 sources/free-exercise-db/preprocess.py
```

### enrich.py â€” LLM enrichment
Calls Claude to enrich exercises with movement patterns, training modalities,
muscle involvements, and unilateral flag. Writes `exercises_enriched.json`.
Run before `ingest.py`. Use `--limit N` and `--random` flags to control scope.

```bash
python3 sources/free-exercise-db/enrich.py --limit 10 --random
```

### ingest.py â€” normalised JSON â†’ RDF
Runs morph-KGC on `exercises_normalized.json`, applies LLM enrichment from
`exercises_enriched.json`, merges ontology vocabulary files, runs repair
queries, and serialises to `sources/free-exercise-db/ingested.ttl`.

```bash
python3 sources/free-exercise-db/ingest.py
```

### validate.py â€” SHACL validation
```bash
python3 sources/free-exercise-db/validate.py
```
Exits 0 if conforms, 1 if violations. Movement pattern constraint relaxed to
`sh:minCount 0` pending Mobility/SoftTissue exemption design (ADR-044, TODO).

**URI conventions (ADR-040):**
- Exercises: `feg:ex_{id}` (avoids leading numeral / invalid NCName issues)
- Involvements: `feg:inv_ex_{id}_{feg_local_name}_{degree}` (no spaces)
